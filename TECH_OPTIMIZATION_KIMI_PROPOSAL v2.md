# 3DMIC-291 + JS6620 舵机跟踪系统技术优化方案

> **版本**：v2.0（综合版）  
> **日期**：2026-06-25  
> **输入文档**：
> - `OPTIMIZATION_DRAFT.md`（以下简称“智谱草案”）
> - `TECH_OPTIMIZATION_KIMI_PROPOSAL v1.md`（以下简称“建议书”）
> **范围**：面向 v1.3 代码，提出可实施的优化方案；不修改代码，仅输出方案。

---

## 1. 目标与现状

### 1.1 优化目标

| 维度 | 当前水平（v1.3） | 目标 |
|---|---|---|
| 响应时间 | 吹哨 0.5 s；语音 0.5–3.9 s（典型），盲区/低 3-mic 帧率时可达 10–15 s | 语音典型响应 ≤ 3 s，极端 ≤ 5 s |
| 指向精度 | 4/5 方位 < ±10°，系统性偏差 6oc +4°、12oc +7°、3oc +16° | 系统性偏差 ≤ ±3°，随机抖动 ≤ ±5° |
| 振荡抑制 | 零振荡（feed-forward 已解决闭环问题） | 保持零振荡，同时提升动态响应 |
| 舵机噪音过滤 | motion-pause 500 ms 单道防线 | 多层级过滤，减少运动期污染与极限嗡鸣 |

### 1.2 当前代码关键路径

```
I²S DMA(100 ms) → deinterleave + pre-emphasis → doa_process
                                               ↓
                     tracker_update → servo_set_angle_deg → LEDC PWM
```

- DOA 帧率：~10 Hz（100 ms 窗）
- `DOA_HIST_N = 5`：lag 中值平滑 500 ms
- `DOA_STABLE_STREAK = 3`：sextant 滞后 300 ms
- `SERVO_MOTION_HOLDOFF_MS = 500`：运动后屏蔽 DOA 500 ms
- `target_agreement_deg = 5°`：需连续 2 帧 3-mic 一致才动作

## 2. 两份输入文档的对比与取舍

### 2.1 共识项（两份文档均认同）

| 优化方向 | 智谱草案 | 建议书 | 综合结论 |
|---|---|---|---|
| 缩短 DMA/分析窗 | 100 ms → 25 ms | 滑动窗 50% 重叠 | **采用滑动窗优先**，25 ms 作为激进备选 |
| 自适应 motion-pause | 按角度差分段 | 按角度差分段 | **采纳**，具体阈值需实测 |
| 校准补偿 | per-sextant 6 float 表 | 全几何标定 | **先实现简单表**，几何标定作为进阶 |
| 前端滤波 | 400 Hz 陷波 | 200 Hz–8 kHz 带通 | **先上陷波**（针对舵机），再评估带通 |
| agreement 逻辑 | 2-of-3 多数 | 置信度加权 EMA | **先实现 2-of-3**，EMA 作为后续 |

### 2.2 分歧项与综合判断

| 议题 | 智谱草案 | 建议书 | 综合判断 |
|---|---|---|---|
| **FFT 1024 → 2048** | 推荐，但指出不能与 25 ms DMA 同时做 | 未重点强调 | **暂缓**。内存/CPU 翻倍，收益仅 ±30% 随机噪声；且 25 ms 窗本身已缩短，FFT 2048 需 42.6 ms 数据，与 25 ms 窗不兼容。建议先测 1024 在 50 ms 重叠下的表现。 |
| **3-TDOA 最小二乘** | 未提及 | 推荐 | **作为 outlier-rejection 手段实现**，不承诺精度大幅提升。三 lag 理想下线相关，残差更适合剔除异常帧。 |
| **PWM 软启动/速度规划** | 未提及 | 推荐 | **采纳**。降低机械冲击和电流噪声，与 motion-pause 配合更好。 |
| **空闲回归 home** | 推荐 | 未提及 | **采纳**。防止舵机长期卡极限。 |
| **速度预测/两层状态机** | 未提及 | 推荐 | **两层状态机采纳**，速度预测对人声价值有限，暂缓。 |
| **β-PHAT / pre-emphasis 调参** | 低优先级 | 未重点 | **均暂缓**。边际收益，调参成本高。 |

### 2.3 对智谱草案中若干判断的修正

1. **“DMA 100 ms → 25 ms 帧率 4×”**：正确，但前提是 CPU 跟得上。40 Hz 下每帧处理时间必须 < 25 ms。当前 3×1024 FFT 约 15 ms（估算），理论上可行，但无余量给日志/WiFi/摄像头。建议先上 **50 ms 滑动重叠（20 Hz）** 验证，再决定是否进一步降到 25 ms。

2. **“DMA 25 ms + FFT 2048 = 120% CPU”**：该估算基于 15 ms/1024 FFT 的假设，但未考虑 ESP32-S3 的 ESP-DSP 库优化。真实值需 `esp_timer_get_time()` 实测。结论应为“**在实测前不建议同时启用**”，而非绝对不可行。

3. **“语音响应 10–15 s → 3–5 s”**：10–15 s 是极端低 3-mic 帧率场景，非典型。主要瓶颈不是 DMA 窗，而是 **3-mic 帧稀缺 + 2-frame agreement**。因此 2-of-3 agreement 比单纯缩窗对语音响应更关键。

4. **“400 Hz 陷波”**：前提需确认舵机噪音主频。JS6620 在不同负载/角度下频率会漂移，固定陷波可能失效。建议先做 UART 录音 + PC FFT 分析，或采用自适应陷波。

## 3. 综合优化方案

### 3.1 响应时间优化

#### 3.1.1 滑动窗 50% 重叠（推荐度 ⭐⭐⭐⭐⭐）

**改造内容**：
- 保持 `MIC_WINDOW_MS = 100`，但每 50 ms 产生一次 DOA 结果。
- 实现方式：在 `mic_task` 中，每次 DMA 读取 50 ms 数据，与前 50 ms 拼接成 100 ms 进行分析。
- 帧率从 10 Hz → **20 Hz**。

**涉及文件**：
- `main/main.c`：调整 DMA 读取循环和 deinterleave 缓冲管理。
- `main/mic_capture.h`：新增 `MIC_WINDOW_MS = 50` 的 DMA 模式，或保留 100 ms 分析窗但 half-step 推进。

**风险**：
- 内存需保存前 50 ms 数据（约 14400 bytes，可接受）。
- CPU 占用从 ~15% → ~30%，仍在安全区。
- 相邻帧高度相关，中值滤波窗口需重新标定。

**预期收益**：
- 控制环延迟从 ~100 ms → ~50 ms。
- `DOA_HIST_N = 5` 对应的时间从 500 ms → 250 ms。
- 对语音响应提升明显，且不牺牲 FFT 分辨率。

#### 3.1.2 Agreement 改为“最近 3 个 3-mic 帧中 2 个一致”（推荐度 ⭐⭐⭐⭐⭐）

**改造内容**：
- 在 `tracker.c` 中，把当前 2-frame 连续一致状态机改为 3-slot ring buffer：
  - 只记录最近 3 个 `DOA_MODE_3MIC` 帧的目标角度。
  - 若其中任意 2 个差值 ≤ `target_agreement_deg`，则视为确认。
  - bad/2-mic 帧不计入，也不清空 buffer。

**涉及文件**：
- `main/tracker.c`：替换 `s_pending_target`/`s_have_pending` 逻辑。

**风险**：
- 噪声场景下可能更快触发误动作；需配合 `min_confidence` 和 out-of-range 门限。
- 多声源切换时可能跟错人。

**预期收益**：
- 语音 3-mic 帧率 16% 时，等待时间从 ~1.5 s → ~0.8 s。
- 与滑动窗配合，典型语音响应可降至 1–2 s。

#### 3.1.3 Lag 中值滤波 5 帧 → 3 帧（推荐度 ⭐⭐⭐⭐）

**改造内容**：
- `doa.h` 中 `DOA_HIST_N 5 → 3`。

**风险**：
- 单帧离群值过滤能力减弱；但 20 Hz 帧率下 3 帧 = 150 ms，仍有一定平滑。
- 建议同时引入“残差门控”（见 3.2.3）作为补充。

**预期收益**：
- 平滑延迟从 500 ms → 250 ms（配合滑动窗）或 300 ms（不重叠）。

#### 3.1.4 自适应 Motion-pause（推荐度 ⭐⭐⭐⭐）

**改造内容**：
- 在 `servo.c` 中，`servo_is_moving()` 的 holdoff 不再固定，而是根据最近一次命令的角度差 `|target - current|` 计算：
  - < 5°：200 ms
  - 5–15°：350 ms
  - > 15°：500 ms
- 同步把 `SERVO_MOTION_HOLDOFF_MS` 改为函数/配置项，不再作为宏。

**涉及文件**：
- `main/servo.h`, `main/servo.c`：重构 motion 状态。
- `main/tracker.c`：调用新的 holdoff 接口。

**风险**：
- 小角度下若舵机余震 > 200 ms，DOA 会拾取残余噪音。需实测。
- 需要记录“命令时刻的实际角度差”，而非仅目标角度。

**预期收益**：
- 小修正响应更快；大运动后仍有足够屏蔽。
- 整体用户体验更“跟手”。

### 3.2 精度优化

#### 3.2.1 Per-sextant 校准补偿表（推荐度 ⭐⭐⭐⭐⭐）

**改造内容**：
- 在 `doa.c` 的 3-mic 分支中，计算 `alpha_deg` 后加上 per-sextant 偏移：
  ```c
  static const float sextant_offset[6] = {+7.0f, 0.0f, 0.0f, +4.0f, 0.0f, 0.0f}; /* 示例 */
  alpha_deg += sextant_offset[out->sextant];
  ```
- 偏移值通过实测 6/12/3/9oc 等方位获得，存入 NVS 或作为常量。

**涉及文件**：
- `main/doa.c`
- 可选 `main/tracker.c` 增加 console 命令 `cal` 用于现场标定。

**风险**：
- 若偏差非系统性（与 SNR、方位有关），固定表效果受限。
- 需要一次认真的标定实验。

**预期收益**：
- 系统性偏差从 ±5–16° 降到 ±2–3°。
- 6 个 float，零运行时开销。

#### 3.2.2 三对 TDOA 残差一致性门控（推荐度 ⭐⭐⭐⭐）

**改造内容**：
- 在 3-mic 求解后，计算三对 TDOA 与理想几何的残差：
  ```text
  residual = |sin(α+60°) + lag_12/K| + |sin(α) - lag_01/K| + |sin(α-60°) - lag_02/K|
  ```
- 若 `residual > threshold`，将该帧降级为 INVALID 或 2-mic。

**涉及文件**：
- `main/doa.c`

**风险**：
- 阈值需标定；过低会误删真信号，过高无效果。
- 9oc 盲区天然 lag_12 不稳定，可能频繁触发降级。

**预期收益**：
- 剔除多径、混响、噪声导致的自相矛盾帧。
- 减少 tracker 收到离谱 azimuth 的概率。

#### 3.2.3 GCC-PHAT 峰值尖锐度门控（推荐度 ⭐⭐⭐）

**改造内容**：
- 在 `gcc_phat()` 中，除 `best_val` 外计算 `mean_val = mean(|R[lag]|)`。
- 增加 `peak_ratio = best_val / mean_val`，仅当 `peak_ratio > threshold`（例如 2.0–3.0）时接受该 lag。

**涉及文件**：
- `main/doa.c`

**风险**：
- 阈值需与现有 `DOA_PEAK_THRESH_*` 联合标定。

**预期收益**：
- 对宽峰/双峰更鲁棒，sub-sample 插值更可靠。
- 随机噪声帧更容易被拒绝。

#### 3.2.4 FFT 1024 → 2048（暂缓，列为 Phase C）

**改造内容**：
- `DOA_FFT_N 1024 → 2048`，窗长需 ≥ 42.6 ms。
- 与 50 ms 滑动窗兼容；与 25 ms DMA 不兼容。

**风险**：
- 内存翻倍（scratch 约 48 KB，仍在 SRAM 可接受范围）。
- CPU 翻倍；若同时 20 Hz 帧率，需实测是否超过预算。
- 所有 peak threshold 需重新标定。

**预期收益**：
- 频率分辨率提高一倍，单帧角度噪声约 ±10° → ±7°。
- 对低频语音谐波分析更有利。

**综合判断**：**先做 3.2.1 和 3.2.2**，它们收益大、风险低。FFT 2048 作为可选项，在 CPU 实测有余量后再启用。

### 3.3 振荡抑制优化

#### 3.3.1 两层跟踪状态机（推荐度 ⭐⭐⭐⭐）

**改造内容**：
- 在 `tracker.c` 中引入：
  - **Acquisition 态**：`|target - current| > 10°` 时，允许较大 step 和较短 agreement 等待。
  - **Tracking 态**：进入 `±10°` 后，切换为小死区、长平滑、小步跟进。

**涉及文件**：
- `main/tracker.c`

**风险**：
- 状态切换边界可能引入抖动；需加 hysteresis（例如进入 Tracking 阈值为 10°，退出为 15°）。

**预期收益**：
- 远距离切换快，近距离稳定。
- 兼顾响应与稳态精度。

#### 3.3.2 PWM 软启动 / 速度规划（推荐度 ⭐⭐⭐⭐）

**改造内容**：
- 在 `servo.c` 中，把一次大角度命令拆成 20 ms 一步的渐变：
  - 每步 PWM 变化 ≤ 5–10 µs。
  - 渐变期间 `servo_is_moving()` 持续为 true。
- 可用简单梯形速度曲线：加速 → 匀速 → 减速。

**涉及文件**：
- `main/servo.c`
- 可能需要 FreeRTOS software timer 或单独 servo task。

**风险**：
- 增加代码复杂度；需确保渐变不阻塞 mic_task。
- 渐变会延长到位时间，需与 adaptive motion-pause 配合。

**预期收益**：
- 降低电机电流冲击和齿轮机械噪声。
- 减少运动期间耦合到麦克风的振动能量。

#### 3.3.3 空闲回归 Home（推荐度 ⭐⭐⭐⭐）

**改造内容**：
- 在 `tracker.c` 中增加 `idle_timer`：
  - 若连续 N 秒（建议 8–12 s）无有效 3-mic 帧，缓慢将舵机移回 0°。
  - 回归速度要慢（例如 2–3°/s），避免突兀。
- 回归期间也进入 motion-pause，避免自身噪音干扰。

**涉及文件**：
- `main/tracker.c`

**风险**：
- 用户短暂沉默后可能误回归，再说话时需重新跟随。
- 若 home 位置不是用户常出现方向，体验反而差。

**预期收益**：
- 防止舵机长期卡在 ±33° 极限。
- 减少极限位置嗡鸣对下次激活的影响。

#### 3.3.4 死区优化（推荐度 ⭐⭐⭐）

**改造内容**：
- 实测 JS6620 + 3.33:1 减速后的最小可分辨角度。
- 将 `deadband_deg` 从 3° 降到 1.5–2°。
- 可选“软死区”补偿：目标超过死区后，命令角度额外加/减死区一半，消除稳态误差。

**涉及文件**：
- `main/tracker.h` 默认配置
- `main/tracker.c` deadband 逻辑

**风险**：
- 死区过小可能导致微小抖动；需实测。

**预期收益**：
- 稳态指向精度提升。

### 3.4 舵机噪音过滤优化

#### 3.4.1 400 Hz（或实测频率）陷波器（推荐度 ⭐⭐⭐⭐）

**改造内容**：
- 在 `main.c` 的 `deinterleave()` 中，pre-emphasis 之后对每个通道应用二阶 IIR 陷波：
  ```c
  /* 设计参数示例：fc=400Hz, Q=5, fs=48000Hz */
  y[n] = b0*x[n] + b1*x[n-1] + b2*x[n-2] - a1*y[n-1] - a2*y[n-2];
  ```
- 陷波系数通过工具预计算（可用 Python scipy.signal.iirnotch）。

**涉及文件**：
- `main/main.c` 的 `deinterleave()`

**风险**：
- 必须先用 UART 录音 + PC FFT 确认舵机噪音主频。若频率漂移（负载/角度相关），固定陷波效果下降。
- 400 Hz 附近若有元音 F1，会有轻微音色损失。
- 改变前端滤波后，需重新标定 `DOA_PEAK_THRESH_*` 和 `lr_independent` 阈值。

**预期收益**：
- 舵机运行时 300–800 Hz 啸叫能量降低 15–25 dB。
- 减少 motion-pause 窗口外的残余污染。

#### 3.4.2 运动期间动态频谱掩蔽（推荐度 ⭐⭐⭐）

**改造内容**：
- 在 `doa.c` 的 `gcc_phat()` 中，当 `servo_is_moving()` 为 true 时，将 300–800 Hz 对应 FFT bins 置零：
  ```c
  if (servo_is_moving()) {
      int lo = 300 * DOA_FFT_N / DOA_FS_HZ;
      int hi = 800 * DOA_FFT_N / DOA_FS_HZ;
      for (int i = lo; i <= hi; i++) {
          s_G[i].re = 0; s_G[i].im = 0;
      }
  }
  ```

**涉及文件**：
- `main/doa.c`

**风险**：
- 由于 tracker 在 motion-pause 期间本就不使用 DOA，此优化主要收益在于：**motion-pause 刚结束时**，DOA 能更快给出干净读数。
- 同样需重新标定 peak threshold。

**预期收益**：
- 缩短有效 settle 时间，可进一步降低 motion-pause 门限。

#### 3.4.3 机械隔振（非代码，但重要）（推荐度 ⭐⭐⭐⭐）

**改造内容**：
- 在舵机与麦克风支架之间加入柔性垫圈（硅胶/泡棉）。
- 避免麦克风 PCB 与舵机支架刚性直接接触。

**风险**：
- 机械改动；用户不能焊接，但垫圈安装应无需焊接。
- 需验证不影响结构强度。

**预期收益**：
- 从物理上降低振动耦合，效果通常优于纯软件滤波。

## 4. 实施阶段

### Phase A：低风险高收益（预计 0.5–1 天）

1. **per-sextant 校准表**（精度，零运行时开销）
2. **agreement 改为 2-of-3**（响应时间，对语音最有效）
3. **lag 中值 5 → 3**（响应时间）
4. **空闲回归 home**（振荡/UX）

**预期结果**：
- 系统性偏差降至 ±3° 以内。
- 语音典型响应从 3.9 s → 1.5–2 s。
- 零振荡保持。

### Phase B：核心架构优化（预计 1–2 天）

5. **滑动窗 50% 重叠（20 Hz DOA）**（响应时间）
6. **自适应 motion-pause**（响应 + 噪音）
7. **PWM 软启动**（噪音 + 振荡）
8. **两层跟踪状态机**（响应 + 振荡）
9. **死区优化**（精度）

**预期结果**：
- 控制延迟降至 50 ms 级。
- 舵机运动更平滑、噪音更低。
- 远近场景都有较好表现。

### Phase C：进阶/验证后启用（预计 1–3 天，视实测而定）

10. **400 Hz 陷波器**（噪音，需先测频率）
11. **三 TDOA 残差门控**（精度/鲁棒性）
12. **GCC 峰值尖锐度门控**（精度）
13. **运动期间动态频谱掩蔽**（噪音）
14. **FFT 1024 → 2048**（精度，需 CPU 实测）
15. **几何标定（Levenberg-Marquardt）**（精度，离线工具）

## 5. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| 滑动窗/高帧率导致 CPU 过载 | 丢帧、DOA 不稳定 | 先用 20 Hz 验证；用 `esp_timer_get_time()` 监控每帧耗时；预留降级到 10 Hz 的开关 |
| 前端滤波改变后 peak threshold 失效 | 3-mic 比例骤降或噪声帧增多 | 每次改滤波都重新标定 `DOA_PEAK_THRESH_3MIC/2MIC` 和 `lr_independent` 阈值 |
| 自适应 motion-pause 过短 | 残余振动污染 DOA | 从小到大逐步降低；保留固定 500 ms 作为 fallback |
| 校准表过拟合特定环境 | 换房间/温度后偏差复现 | 把校准表存 NVS，并提供 console 命令重新标定；考虑温度补偿扩展 |
| PWM 软启动延长到位时间 | 响应变慢 | 软启动参数可调；小角度命令跳过软启动 |
| 陷波器频率不准 | 语音损失或噪音未滤除 | 先用 PC FFT 分析录音；或采用自适应陷波 |

## 6. CPU / 内存预算

| 配置 | 帧率 | FFT/帧 | 预估 CPU | 内存增量 | 备注 |
|---|---|---|---|---|---|
| 当前 v1.3 | 10 Hz | 3×1024 | ~15% | — | 基准 |
| Phase A | 10 Hz | 3×1024 | ~15% | +24 bytes（校准表）| 无显著负载 |
| + 50% 滑动窗 | 20 Hz | 3×1024 | ~30% | +14 KB（前 50 ms 缓冲）| 推荐 |
| + 25 ms DMA | 40 Hz | 3×1024 | ~60% | +14 KB | 激进，余量小 |
| + FFT 2048 | 20 Hz | 3×2048 | ~60% | +24 KB | 与 50 ms 窗兼容 |
| + 25 ms + FFT 2048 | 40 Hz | 3×2048 | ~120% | +38 KB | **不建议**，除非隔帧处理 |

**结论**：
- 推荐配置：**50% 滑动窗 + 1024 FFT + Phase A 全部**（CPU ~30%）。
- 激进配置：**25 ms DMA + 1024 FFT**（CPU ~60%），仅在不跑 WiFi/摄像头时考虑。
- FFT 2048 与 25 ms DMA 不建议同时启用。

## 9. 结论

结合智谱草案的“快速量化优化”与建议书的“控制理论深度”，本方案提出一条**先低风险高收益、再核心架构、最后进阶验证**的三阶段路线：

- **Phase A** 可在半天内完成，主要解决精度和语音响应问题；
- **Phase B** 在 1–2 天内将系统从 10 Hz 控制环升级到 20 Hz，并降低运动噪音；
- **Phase C** 作为可选进阶，在实测 CPU/内存余量后逐步启用。

最关键的三项改动是：**校准表、2-of-3 agreement、滑动窗**。它们共同作用，可在不改变硬件的前提下，把语音响应从数秒级降到 1–2 秒级，同时保持甚至提升指向精度，并维持零振荡。
