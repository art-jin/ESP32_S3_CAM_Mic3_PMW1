# 优化计划讨论草稿

> **状态**：草稿，待讨论
> **日期**：2026-06-24
> **基于**：v1.3（feed-forward + 前置预加重 + 270° 舵机修正 + ±33° clamp）
> **目的**：从响应时间、精度、振荡抑制、舵机噪音过滤四个维度，记录可优化点和预估收益，供下次讨论后决定实施哪些。

---

## 一、响应时间

**当前**：吹哨 0.5s，语音 10-15s（几何盲区位置）

### 延迟分解

| 环节 | 当前延迟 | 可优化 | 说明 |
|---|---|---|---|
| DMA 窗口 | 100ms | ✅ **最大瓶颈** | 每帧抓 4800 样本/通道，FFT 只用前 1024（21%），其余浪费 |
| Lag 中位数滤波 | 500ms | ✅ | 5 帧 × 100ms/帧 |
| target_agreement | 1-1.5s（语音）| ✅ | 要求连续 2 帧 3-mic 一致；语音帧率 16% |
| Motion-pause | 500ms | ✅ | `SERVO_MOTION_HOLDOFF_MS=500` |
| 舵机机械时间 | 200-400ms | ❌ | JS6620 物理特性 |

### 优化方案

#### ① DMA 窗口 100ms → 25ms（影响最大，推荐度 ⭐⭐⭐⭐⭐）

当前：抓 4800 样本/通道 → 只用前 1024 做 FFT → 剩 3776 样本**浪费**。

改为 25ms 窗口（1200 样本/通道）：
- 帧率 10 Hz → **40 Hz**
- FFT 仍用 1024 点（21.3ms 数据足够）
- 中位数滤波 500ms → **125ms**（5 帧 × 25ms）
- CPU 从 ~15% → ~60%（4× 帧率），仍可接受
- **语音响应预估：10-15s → 3-5s**

改动文件：`mic_capture.h` 的 `MIC_WINDOW_MS` + `MIC_WINDOW_BYTES` + `MIC_WINDOW_SAMPLES`

代价：
- 每帧 GCC-PHAT 处理量不变，但调用频率 4×
- CPU 占用增加，可能需要降低日志频率
- 如果同时做 FFT 2048（见下文），CPU 可能到 80-90%，需要实测

#### ② Lag 中位数 5 帧 → 3 帧（推荐度 ⭐⭐⭐⭐）

减少 200ms 平滑延迟。3 帧中位数仍能过滤单帧离群值。

改动文件：`doa.h` 的 `DOA_HIST_N`

代价：抗噪稍差（但 feed-forward 已经提供了系统级稳定性，lag 噪声只影响单帧 az 精度）。

#### ③ Motion-pause 500ms → 300ms（推荐度 ⭐⭐⭐）

JS6620 机械稳定时间约 200ms，300ms 有 100ms 安全余量。

改动文件：`servo.h` 的 `SERVO_MOTION_HOLDOFF_MS`

代价：如果舵机余震 > 300ms（廉价舵机可能），motion-pause 结束后 DOA 可能拾到残余噪音。需要实测。

#### ④ Agreement 从"连续 2 帧"改为"最近 3 帧中 2 帧一致"（推荐度 ⭐⭐⭐）

当前：要求 2 个**连续** 3-mic 帧在 ±5° 内。如果中间夹 1 个 2-mic 或 bad 帧，streak 清零。

改为：最近 3 个 3-mic 帧中只要有 2 个在 ±5° 内即可。

对语音（3-mic 帧率 16%）尤其有效：
- 当前等待：~1.5s（需要连续 2 个 3-mic 帧）
- 改后等待：~0.8s（允许中间有 1 个噪声帧）

改动文件：`tracker.c` 的 agreement 逻辑，从 2-frame state machine 改为 3-frame ring buffer + majority check

### 响应时间优化预估

| 配置 | 吹哨响应 | 语音响应 |
|---|---|---|
| 当前 v1.3 | 0.5s | 10-15s |
| + DMA 25ms + median 3 + pause 300ms | ~0.15s | 3-5s |
| + agreement 2-of-3 | ~0.15s | 1-2s |

## 二、精度

**当前**：±5°（6oc），±16°（3oc），±36°（9oc 盲区 2-mic fallback）

### 精度瓶颈

#### ① K = 1.4 样本（根本物理限制）

K = d·fs/c = 0.01 × 48000 / 343 = 1.40 样本。

- 抛物线插值精度 ±0.3 样本 → 占 K 的 21% → 单帧角度噪声 ±10°
- 不是算法问题，是 10mm 间距 + 48kHz 的**物理极限**
- 提高 K 需要：间距 ↑（改硬件）、采样率 ↑（PDM2PCM 可能不支持 96kHz）、声速 ↓（不现实）

#### ② FFT 1024 点 → 2048 点（推荐度 ⭐⭐⭐）

| 项 | 1024 点（当前）| 2048 点 |
|---|---|---|
| 频率分辨率 | 46.9 Hz/bin | 23.4 Hz/bin |
| 时间窗口 | 21.3ms | 42.6ms |
| 单帧角度噪声 | ±10° | **±7°**（预估）|
| FFT scratch 内存 | ~24KB | ~48KB（SRAM 可放）|
| 计算时间 | ~15ms/帧（3 对）| ~30ms/帧 |
| 语音友好度 | 刚好覆盖 1 个辅音 | **覆盖 1-2 个音素** |

改动文件：`doa.h` 的 `DOA_FFT_N`、`DOA_FFT_LOGN`，以及 twiddle factor / scratch 数组大小

代价：内存翻倍（PSRAM 有 8MB 备用），CPU 翻倍（如果同时做 DMA 25ms，总 CPU 可能到 80%）。

#### ③ 校准补偿表（推荐度 ⭐⭐⭐⭐⭐）

当前无校准补偿。实测偏差：
- 6oc: +4°
- 12oc: +7°
- 3oc: +16°（可能含几何盲区效应）

加 per-sextant 校准表可以消除系统性偏差：
```c
static const float cal_offset[6] = {+7, 0, 0, +4, 0, 0};
azimuth_corrected = azimuth_raw + cal_offset[sextant];
```

效果：绝对精度 ±5° → **±2°**（假设偏差是固定 systematic error）。

代价：6 个 float 常量，零计算开销。

改动文件：`doa.c` 的 `doa_process` 在计算 alpha_deg 后加一行查表。

#### ④ β-PHAT（部分白化）（推荐度 ⭐⭐）

当前标准 PHAT（完全白化）：`G[k] = cross[k] / |cross[k]|`

替代 β-PHAT：`G[k] = |cross[k]|^β · exp(iφ[k])`，β=0.8

保留部分幅度信息，在低 SNR 下更鲁棒（高能量频率 bin 权重更高）。

代价：需调 β 参数（0.5-1.0 扫描），且 `powf()` 计算开销。

#### ⑤ Pre-emphasis 系数调参（推荐度 ⭐⭐）

当前 0.97（教科书标准）。这是电话语音优化的值，不一定适合 10mm 麦阵。

扫描 0.90-0.99，找到 ρ01（L/R 相关性）和 peak01（GCC-PHAT 峰值）的最优分离点。

代价：烧多次固件实测。

### 精度优化预估

| 优化 | 当前 | 改善后 | 代价 |
|---|---|---|---|
| 校准表 | ±5° 绝对 | ±2° 绝对 | 6 个数字 |
| FFT 2048 | ±10°/帧 | ±7°/帧 | 2× 内存 + CPU |
| β-PHAT | ±10°/帧 | ±8°/帧 | 调参 + powf |
| Pre-emphasis 调参 | — | 边际 | 多次烧录 |

## 三、振荡抑制

**当前**：零振荡（feed-forward 补偿 + ±33° clamp + agreement + motion-pause）

### 潜在隐患

#### ① Feed-forward 使用命令角度而非实际角度

```c
alpha_room = az_array + servo_get_angle_deg();  // 命令值，非实际值
```

JS6620 有 1-3° 齿轮回差 → disc 端 0.3-0.9°。feed-forward 有微小系统性误差。

不会导致振荡（误差太小），但可能引入固定偏差。

**优化**：读舵机电位器（需拆线引出 ADC 引脚）。当前硬件不支持，优先级低。

#### ② 空闲时舵机卡在极限（推荐度 ⭐⭐）

用户离开后不再说话 → 无 3-mic 帧 → tracker 不更新 → 舵机停在 ±33°。

**优化**：加"空闲回归"策略：
- > 10 秒无 3-mic 帧 → 缓慢回到 home (0°)
- 用 PWM 渐变而非跳变（避免突兀运动）

改动文件：`tracker.c` 加 idle_timer 状态

#### ③ 多声源跳变（推荐度 ⭐）

两人说话时 GCC-PHAT 峰值跳变 → tracker 跟着跳。

**优化**：加"源连续性"检查：
- 新 target 与上次差 > 30° → 要求 3 帧一致（而非 2 帧）
- 代价：切换说话人时响应变慢

#### ④ 舵机特定角度谐振（推荐度 ⭐）

某些 PWM 占空比下舵机控制环谐振。需实测扫描找谐振点。

**优化**：在软件中避开谐振角度（跳过特定 PWM 值）。

## 四、舵机噪音过滤

**当前**：motion-pause（500ms）是唯一防线

### 当前问题

| 问题 | 严重度 | 说明 |
|---|---|---|
| 只在运动期间过滤 | 中 | 舵机保持极限位置时嗡鸣不在 motion-pause 窗口内 |
| Pre-emphasis 不滤舵机噪音 | 低 | 预加重截止 ~229Hz，舵机噪音 300-800Hz 在通带内 |
| 无频域过滤 | 中 | 时域全频段处理 |

### 优化方案

#### ① 陷波器 Notch filter 400Hz（推荐度 ⭐⭐⭐⭐）

在预加重之后加一个二阶 IIR 陷波，针对舵机电机主要频率：

```
设计参数：fc=400Hz, Q=5, fs=48kHz
y[n] = b0·x[n] + b1·x[n-1] + b2·x[n-2] - a1·y[n-1] - a2·y[n-2]
```

- 陷波深度：-20dB 以上
- 对语音影响：400Hz ± 40Hz 有元音第一共振峰 F1，但带宽窄，影响小
- 计算量：每样本每通道 5 乘法 + 4 加法，可忽略

改动文件：`main.c` 的 `deinterleave()` 中，pre-emphasis 之后加 notch

需要先实测确认舵机噪音的准确频率（可能不是 400Hz，需用 PC 端 FFT 分析 UART 录音数据）。

#### ② 自适应 motion-pause（推荐度 ⭐⭐⭐）

不是固定 500ms，根据移动量动态调整：
- 移动 < 5° → pause 200ms（快）
- 移动 5-20° → pause 400ms
- 移动 > 20° → pause 800ms（慢，大角度运动后余震长）

改动文件：`servo.c` 的 `servo_is_moving()` 改为基于移动量计算 holdoff

#### ③ 动态频谱掩蔽（推荐度 ⭐⭐⭐）

在 GCC-PHAT 的频域，**只在 servo_is_moving() 期间**把舵机噪音频率 bins 置零：

```c
if (servo_is_moving()) {
    int bin_lo = 300 * DOA_FFT_N / DOA_FS_HZ;  // ≈ bin 6
    int bin_hi = 800 * DOA_FFT_N / DOA_FS_HZ;  // ≈ bin 17
    for (int i = bin_lo; i <= bin_hi; i++) {
        s_G[i].re = 0;
        s_G[i].im = 0;
    }
}
```

- 优点：运动时完全消除 300-800Hz 干扰
- 缺点：运动时语音 F1 信息丢失（但运动期间 tracker 不更新，所以不影响跟踪）
- 改动文件：`doa.c` 的 `gcc_phat()` 中

#### ④ 参考麦自适应降噪 LMS（推荐度 ⭐）

在舵机附近放第 4 颗麦，专门拾取舵机噪音。用 LMS/RLS 从 3 颗主麦中减去。

- 效果最好（-30dB+）
- 需要：第 4 颗麦 + S3 多一个 I²S DIN → **硬件改动**
- 复杂度：~200 行代码

## 六、实施建议（如果决定做）

1. 校准补偿表（#2）
2. Lag 中位数 5→3（配合 DMA 缩短）
3. Motion-pause 500→300ms

### 阶段 B：核心优化（1 天）

### 阶段 C：进阶优化（1-2 天，可选）
7. FFT 2048（需评估 CPU + DMA 25ms 的组合是否过载）
8. 自适应 motion-pause

- 参考麦 LMS（硬件改动太大）

## 七、CPU 预算分析

当前（v1.3，DMA 100ms，FFT 1024）：
- 帧率 10 Hz
- 每帧 3 对 GCC-PHAT × 1024pt FFT = ~15ms
- CPU 占用：15ms / 100ms = **15%**

优化后预估：

| 配置 | 帧率 | FFT/帧 | CPU 占用 | 可行？ |
|---|---|---|---|---|
| 当前 | 10 Hz | 3×1024 = 15ms | 15% | ✓ |
| DMA 25ms only | 40 Hz | 3×1024 = 15ms | 60% | ✓ |
| DMA 25ms + FFT 2048 | 40 Hz | 3×2048 = 30ms | **120%** | ✗ 超载 |
| DMA 25ms + FFT 2048 + 跳帧 | 40 Hz 读 / 20 Hz 算 | 3×2048 = 30ms | 60% | ✓ 但需改架构 |
| DMA 50ms + FFT 2048 | 20 Hz | 3×2048 = 30ms | 60% | ✓ 折中 |

**结论**：DMA 25ms 和 FFT 2048 **不能同时做**，除非改为"隔帧处理"（DMA 抓 40Hz 但 DOA 只算 20Hz）。

折中方案：DMA 50ms（20Hz 帧率）+ FFT 2048，CPU ~60%，响应时间仍比当前快 2×。
